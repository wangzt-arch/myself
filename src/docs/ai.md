# 从 0 开始使用 Codex

## Codex 是什么

Codex 是一个面向代码工作的 AI 协作助手。它可以阅读项目、修改文件、运行命令、启动本地服务、检查页面效果、解释代码、排查错误，也可以帮助你把一个模糊需求拆成可执行的开发步骤。

更准确地说，Codex 不是单纯的聊天机器人，而是一个可以参与工程流程的开发助手。

## 适合让 Codex 做什么

Codex 适合处理这些任务：

- 分析项目结构和技术栈
- 修复 bug
- 重构组件
- 优化页面 UI 和交互
- 编写或调整配置
- 补充文档
- 运行构建和测试
- 查看本地页面效果
- 根据报错定位问题
- 解释一段代码为什么这样写

不太适合直接交给 Codex 的任务：

- 没有上下文的大范围重写
- 没有验收标准的“随便优化”
- 涉及真实账号、支付、生产数据的高风险操作
- 需要长期产品决策的模糊判断

## 第一次使用 Codex 的推荐流程

### 1. 先说目标

不要只说“优化一下”。更好的表达方式是：

```txt
优化 Docs 模块，让它支持搜索、分类、目录和代码复制。
```

或者：

```txt
模型预览页面第一次进入默认模型显示不完整，请让模型自动适配视野。
```

目标越清楚，Codex 越容易做出可验证的改动。

### 2. 指定范围

如果只想改某个模块，可以直接说明：

```txt
只改 src/pages/docs，不要动其它页面。
```

如果可以顺手清理相关问题，也可以说明：

```txt
可以顺手修掉相关 warning，但不要升级大版本依赖。
```

### 3. 给出验收标准

验收标准可以很简单：

```txt
构建通过，页面没有控制台 error，移动端不遮挡。
```

或者：

```txt
第一次进入 /preview 时能直接看到默认模型。
```

### 4. 让 Codex 实施

可以直接说：

```txt
动手吧。
```

Codex 会读取相关文件、修改代码、运行必要验证，并汇报结果。

## 一个完整需求示例

```txt
优化 Docs 页面。
要求：
1. 左侧文档按分类展示。
2. 支持搜索标题、标签和正文。
3. 右侧显示当前文章目录。
4. 代码块有复制按钮。
5. 构建通过，浏览器没有 error。
```

这个需求有目标、范围、功能点和验收标准，执行起来会更稳定。

## 使用 Codex 的沟通技巧

### 把问题描述成现象

不要只说：

```txt
页面坏了。
```

更好：

```txt
进入 /preview 后画布是空的，选择下拉框后模型才出现。
```

### 把期望描述清楚

不要只说：

```txt
好看一点。
```

更好：

```txt
首页要像个人技术作品集，第一屏有介绍、主按钮和项目入口，不要营销风。
```

### 分阶段推进

大需求建议拆成几轮：

1. 先把功能做通。
2. 再优化交互。
3. 再做视觉细节。
4. 最后做性能和构建质量。

这样风险更低，也更容易回滚。

## Codex 常用能力

### 读代码

Codex 可以快速定位：

- 路由入口
- 页面组件
- 样式文件
- 配置文件
- 报错来源
- 未使用代码

### 改代码

Codex 可以直接修改项目文件。比较适合：

- React 组件
- CSS / SCSS 样式
- Webpack 配置
- Markdown 文档
- 工具函数

### 跑命令

常见命令：

```bash
npm start
npm run build
npm test
```

如果命令失败，Codex 会根据日志继续定位。

### 看浏览器

在前端项目里，浏览器验证很重要。Codex 可以：

- 打开本地页面
- 查看控制台日志
- 点击按钮
- 输入搜索词
- 截图检查布局
- 验证移动端宽度

## AI 基础知识

## 大模型是什么

大模型是一种通过大量文本、代码和其它数据训练出来的预测模型。它的核心能力是根据上下文生成最可能有用的输出。

它并不是数据库，也不是绝对可靠的事实来源。它更像一个强大的模式识别和推理工具。

## Prompt 是什么

Prompt 是你给 AI 的任务说明。好的 prompt 通常包含：

- 背景
- 目标
- 输入
- 输出格式
- 约束条件
- 验收标准

示例：

```txt
请分析这个 React 页面性能问题。
只关注首屏渲染和包体积。
输出按“问题、原因、建议改法”组织。
```

## 上下文是什么

上下文是 AI 当前能看到的信息，包括：

- 你的问题
- 项目文件内容
- 命令输出
- 浏览器页面状态
- 之前的对话

上下文越准确，结果越可靠。上下文错误或缺失时，AI 可能会猜。

## AI 的常见风险

### 幻觉

AI 可能生成看似合理但实际不存在的 API、配置或事实。

应对方式：

- 让它引用具体文件
- 运行构建或测试
- 用官方文档确认不稳定信息

### 过度改动

AI 可能为了完成目标改动过大。

应对方式：

```txt
只做最小改动，不要重构无关文件。
```

### 忽略边界条件

比如只在桌面端看起来正常，移动端却遮挡。

应对方式：

```txt
同时检查桌面和移动端。
```

## AI 编程的推荐工作流

### 1. 让 AI 先读代码

```txt
先分析相关文件，不要立刻修改。
```

### 2. 让 AI 做小步改动

```txt
先实现搜索功能，暂时不要做视觉重构。
```

### 3. 让 AI 验证结果

```txt
改完后跑 npm run build，并打开页面检查控制台。
```

### 4. 人来做最终判断

AI 可以提高效率，但最终的产品判断、代码风格取舍和上线风险仍然应该由开发者负责。

## 在这个项目中可以继续做的 AI 方向

### AI 文档助手

基于 Docs 模块做一个问答入口：

- 输入问题
- 在当前文档中检索相关内容
- 生成答案
- 给出引用位置

### AI 翻译增强

当前项目已有翻译页面，可以扩展：

- 多语言互译
- 技术术语优化
- Markdown 翻译
- 翻译历史

### AI 代码解释

给文档或代码片段增加解释能力：

- 解释代码用途
- 输出关键点
- 生成示例
- 提醒潜在问题

### AI 项目总结

为个人站增加项目总结功能：

- 根据已有页面生成作品介绍
- 自动提炼技术栈
- 生成 README
- 生成简历项目描述

## 最小实践建议

刚开始用 Codex 时，不要追求一次完成所有事情。推荐按这个顺序练习：

1. 让 Codex 解释一个页面。
2. 让 Codex 修一个小 bug。
3. 让 Codex 优化一个组件。
4. 让 Codex 补一篇文档。
5. 让 Codex 做一次完整功能改造。

每一步都要求它验证结果。这样你会更快建立对 AI 协作边界的判断。
