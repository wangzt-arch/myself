# AI 开发提示语最佳实践

## 文档目标

这篇文档整理 AI 开发中高频、稳定、可复用的提示语写法。重点不是堆“神奇咒语”，而是形成一套可复制的表达模板，让 AI 更容易理解目标、边界、上下文和验收标准。

适用场景：

- 使用 Codex 修改代码
- 使用 ChatGPT 分析问题
- 使用 AI 生成文档
- 使用 AI 做方案设计
- 使用 AI 排查 bug
- 使用 AI 辅助学习

## 核心原则

## 1. 说清楚目标

差的提示语：

```txt
优化一下这个页面。
```

更好的提示语：

```txt
优化 Docs 页面，让它支持分类导航、搜索、文章目录和代码块复制。改完后运行 npm run build，并检查页面没有控制台 error。
```

目标要包含：

- 要做什么
- 修改范围
- 期望结果
- 验证方式

## 2. 给足上下文

差的提示语：

```txt
这个报错怎么解决？
```

更好的提示语：

```txt
这个 React 项目启动时报错，报错信息如下。请先判断错误来源，再给出最小修改方案，不要直接升级依赖。

报错：
...
```

AI 不是项目负责人，它需要上下文才能减少猜测。

## 3. 明确边界

常用边界话术：

```txt
只修改当前模块，不要改无关文件。
```

```txt
不要升级依赖，除非没有其它解决方案。
```

```txt
不要改变现有路由和接口协议。
```

```txt
保持现有 UI 风格，不要重做设计系统。
```

```txt
如果需要大范围重构，先说明方案，不要直接动手。
```

## 4. 要求可验证

推荐加上验收标准：

```txt
完成后需要满足：
1. npm run build 通过。
2. 页面没有 console error。
3. 移动端宽度下没有遮挡。
4. 原有功能不受影响。
```

没有验收标准，AI 很容易只给“看起来差不多”的结果。

## 5. 分阶段推进

大任务不要一次性丢给 AI。

推荐：

```txt
先分析问题，不要修改代码。
```

```txt
先实现功能，样式后面再优化。
```

```txt
先做最小可用版本，通过后再补交互细节。
```

## 通用提示语模板

## 任务执行模板

```txt
你是一个资深前端工程师。

目标：
【写清楚要完成什么】

背景：
【描述当前项目、页面、问题现象】

要求：
1. 【具体要求 1】
2. 【具体要求 2】
3. 【具体要求 3】

约束：
1. 只修改【范围】。
2. 不要改动无关文件。
3. 不要升级依赖。

验收：
1. npm run build 通过。
2. 页面没有控制台 error。
3. 【业务验收标准】

请直接实施。
```

## 问题排查模板

```txt
请排查这个问题。

现象：
【用户看到的问题】

复现步骤：
1. 打开【页面/功能】
2. 点击【操作】
3. 出现【结果】

期望结果：
【正确行为】

限制：
先定位原因，再做最小修改。
不要重构无关模块。
```

## 代码审查模板

```txt
请 review 这次改动。

重点关注：
1. 是否有运行时 bug。
2. 是否破坏现有功能。
3. 是否有边界条件遗漏。
4. 是否有不必要的复杂度。
5. 是否缺少必要验证。

输出格式：
先列问题，按严重程度排序。
如果没有问题，明确说明没有发现阻塞问题。
```

## 重构模板

```txt
请重构这个模块。

目标：
提升可维护性，不改变现有功能。

要求：
1. 保持外部 API 不变。
2. 保持页面行为不变。
3. 优先拆分职责，不要过度抽象。
4. 重构后运行构建验证。

如果发现需要改变行为，先说明原因。
```

## UI 优化模板

```txt
请优化这个页面的 UI。

定位：
这是一个【工具型 / 文档型 / 作品集 / 后台管理】页面。

目标用户：
【开发者 / 普通访客 / 管理人员】

设计要求：
1. 页面信息层级更清晰。
2. 移动端不遮挡。
3. 按钮、列表、卡片尺寸稳定。
4. 不使用过度装饰。
5. 保持当前项目视觉风格。

验收：
打开页面截图检查，控制台无 error。
```

## 文档生成模板

```txt
请为这个功能写一篇 Markdown 文档。

内容包括：
1. 功能简介
2. 使用场景
3. 操作步骤
4. 常见问题
5. 最佳实践

要求：
语言简洁，偏实用，不写营销文案。
```

## Codex 开发常用话术

## 启动项目

```txt
启动这个前端项目。先检查 package.json 的 scripts，再用正确命令启动。启动后告诉我访问地址。
```

## 分析项目

```txt
分析这个项目的技术栈、目录结构、主要路由和启动方式。只分析，不修改代码。
```

## 修复明确 bug

```txt
修复【页面/模块】中的这个问题：
【描述问题现象】

要求：
1. 先定位原因。
2. 做最小修改。
3. 保持现有功能不受影响。
4. 改完后验证。
```

## 添加功能

```txt
给【模块】添加【功能】。

具体要求：
1. 【功能点 1】
2. 【功能点 2】
3. 【功能点 3】

约束：
遵循当前代码风格，不引入新依赖，除非有明确必要。
```

## 处理控制台报错

```txt
检查当前页面的浏览器控制台 error/warning。先说明每个问题的来源，再处理会影响运行或体验的问题。
```

## Git 提交

```txt
把当前改动提交并推送到 master。

要求：
1. 先检查 git status。
2. 只提交本次相关文件。
3. commit message 使用 Conventional Commits。
4. 推送后确认本地和远程同步。
```

## 常用高质量话术

## 让 AI 先问问题

```txt
如果信息不足，请先问我最多 3 个关键问题，不要直接猜。
```

适合需求不明确时使用。

## 让 AI 先列方案

```txt
先给出 2-3 个可行方案，说明优缺点和推荐方案。暂时不要修改代码。
```

适合架构调整、技术选型、复杂重构。

## 控制改动范围

```txt
只做完成这个需求所需的最小改动。
```

适合修 bug。

## 要求保护已有改动

```txt
当前工作区可能有我手动修改的内容。不要回滚或覆盖你没有明确修改过的文件。
```

适合多人协作或工作区不干净时。

## 要求解释原因

```txt
修改完成后说明：
1. 根因是什么。
2. 改了哪些文件。
3. 为什么这样改。
4. 如何验证。
```

适合需要学习或复盘时。

## 要求保守实现

```txt
优先使用项目现有模式，不新增抽象，不引入新依赖。
```

适合老项目维护。

## 要求面向生产

```txt
请按生产可维护标准实现，关注错误处理、边界状态、可访问性和移动端表现。
```

适合核心功能。

## 不同场景的提示语

## 学习一个概念

```txt
用前端开发者能理解的方式解释【概念】。
要求：
1. 先给一句话定义。
2. 再给实际例子。
3. 最后说明常见误区。
```

## 生成学习计划

```txt
我想学习【主题】。
当前水平：【水平】
目标：【目标】
时间：【周期】

请给我一个可执行学习计划，每天包含学习内容和练习任务。
```

## 让 AI 写代码

```txt
请实现【功能】。

技术栈：
【React/Vue/Node 等】

要求：
1. 代码可运行。
2. 保持类型/状态清晰。
3. 处理 loading、empty、error 状态。
4. 给出必要注释。
```

## 让 AI 优化代码

```txt
请优化这段代码。

目标：
1. 提高可读性。
2. 降低重复逻辑。
3. 不改变行为。

请先指出问题，再给出修改后的代码。
```

## 让 AI 排查性能

```txt
分析这个页面的性能问题。

重点看：
1. 首屏加载。
2. 包体积。
3. 不必要渲染。
4. 大资源加载。

输出：
问题、原因、优化建议、优先级。
```

## 让 AI 生成测试用例

```txt
请为这个函数/组件设计测试用例。

要求覆盖：
1. 正常输入。
2. 边界输入。
3. 异常输入。
4. 空状态。

先列测试点，再写测试代码。
```

## 让 AI 改文案

```txt
请优化这段产品文案。

要求：
1. 更清晰。
2. 更短。
3. 不夸张。
4. 保持专业语气。

输出 3 个版本，并说明适用场景。
```

## 提示语结构建议

## 推荐结构

```txt
角色：
你是【角色】。

任务：
【要做什么】。

上下文：
【项目、页面、数据、限制】。

要求：
【具体要求列表】。

输出：
【期望格式】。

验收：
【如何判断完成】。
```

## 不推荐结构

```txt
帮我看看。
```

```txt
随便优化一下。
```

```txt
写得高级一点。
```

这些提示语缺少目标和边界，容易得到不可控结果。

## AI 开发中的最佳实践

## 小步提交

一次只让 AI 做一个明确任务。每完成一步就验证。

推荐：

```txt
先只实现搜索功能，不处理样式。
```

不推荐：

```txt
把整个系统都优化一下。
```

## 明确不要做什么

告诉 AI 不要做的事，通常和告诉它要做什么一样重要。

示例：

```txt
不要改接口字段。
不要新增依赖。
不要改路由。
不要重写整个页面。
```

## 要求保留现有行为

```txt
新增功能时保留已有交互，不要影响原有入口。
```

## 让 AI 自检

```txt
改完后请自检：
1. 是否有未使用变量。
2. 是否有控制台报错。
3. 是否有移动端遮挡。
4. 是否有无关文件变更。
```

## 让 AI 汇报验证结果

```txt
最终回复请包含：
1. 修改了什么。
2. 验证了什么。
3. 还有什么风险。
```

## 常见反模式

## 过度依赖“角色扮演”

```txt
你是世界顶级专家……
```

角色描述可以用，但不能替代清晰需求。更重要的是目标、上下文和验收标准。

## 只要求“详细”

```txt
详细讲讲。
```

更好的方式：

```txt
按“定义、原理、例子、常见误区、实践建议”解释。
```

## 让 AI 一次性做太多

```txt
重构项目、优化性能、修 bug、加功能、顺便美化页面。
```

这种任务风险高，结果不可控。应该拆分。

## 没有验证要求

```txt
改一下。
```

更好的方式：

```txt
改完后运行 npm run build，并打开页面检查控制台。
```

## 可直接复制的提示语清单

## 开始分析

```txt
先分析相关代码，不要修改。请说明模块结构、关键状态、数据流和潜在问题。
```

## 最小修复

```txt
请做最小修复，只解决当前问题，不做无关重构。
```

## 安全重构

```txt
请在不改变外部行为的前提下重构这段代码。改完后说明行为为什么保持一致。
```

## 页面验证

```txt
请打开本地页面验证：
1. 页面是否正常渲染。
2. 交互是否可用。
3. 控制台是否有 error/warning。
```

## 生成文档

```txt
请把这次功能改动整理成 Markdown 文档，包含背景、用法、实现思路和常见问题。
```

## 生成 commit

```txt
请根据当前改动生成规范 commit message，使用 Conventional Commits。
```

## 总结

好的提示语不是越长越好，而是要做到：

- 目标明确
- 上下文充分
- 范围清晰
- 输出可控
- 验收可验证

当 AI 的输出不稳定时，不要急着换模型。先检查提示语里是否缺少目标、上下文、约束或验收标准。
