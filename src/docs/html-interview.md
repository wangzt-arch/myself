# HTML 面试题

> 前端面试中 HTML 相关的高频问题，以口语化回答为主，由浅入深排列。

---

## 一、基础概念

### 1.1 HTML5 有哪些新特性？

HTML5 是 HTML 的第五次重大升级，带来了语义化标签、多媒体支持、本地存储、Canvas 绘图、WebSocket 等很多新能力。

**语义化标签**是最常用的，比如 `header`、`nav`、`main`、`article`、`section`、`aside`、`footer` 这些标签，让页面结构更清晰，对搜索引擎和屏幕阅读器更友好。

**多媒体**方面，HTML5 原生支持 `video` 和 `audio` 标签，不需要再依赖 Flash 插件。

**表单增强**方面，新增了很多输入类型，比如 `email`、`tel`、`date`、`range`、`color` 等，浏览器会自动做基本的格式验证。

**本地存储**方面，提供了 `localStorage` 和 `sessionStorage`，可以存储键值对数据，容量比 Cookie 大得多。

**其他重要特性**还有 Canvas 2D 绘图、WebSocket 实时通信、Geolocation 地理定位、Web Worker 后台线程、History API 路由控制等。

---

### 1.2 什么是语义化？为什么要语义化？

语义化就是**用合适的 HTML 标签来表达内容的含义**，而不是全部用 `div` 和 `span` 堆砌。

比如页面头部用 `header`，导航用 `nav`，主要内容用 `main`，文章用 `article`，侧边栏用 `aside`，页脚用 `footer`。

语义化的好处主要有三个：

第一，**对 SEO 友好**。搜索引擎爬虫能更好地理解页面结构，知道哪些是标题、哪些是正文、哪些是导航，有助于提升搜索排名。

第二，**对无障碍访问友好**。屏幕阅读器会根据标签类型给视障用户读出不同的提示，比如读到 `nav` 会告诉用户这是导航区域。

第三，**代码可读性更好**。开发者一眼就能看出页面的结构层次，维护起来更方便。

---

### 1.3 `DOCTYPE` 的作用是什么？

`DOCTYPE` 是文档类型声明，放在 HTML 文件的最开头，用来告诉浏览器当前页面使用哪个版本的 HTML 规范来解析。

HTML5 的声明非常简单，就是 `<!DOCTYPE html>`。而在 HTML4 时代，声明非常复杂，需要写一大串 DTD 地址。

如果没有 `DOCTYPE` 声明，浏览器会进入**怪异模式（Quirks Mode）**，用兼容旧浏览器的方式来解析页面，这会导致很多 CSS 布局问题。有了正确的 `DOCTYPE`，浏览器就会进入**标准模式（Standards Mode）**，按照 W3C 规范来渲染页面。

---

### 1.4 `script` 标签中 `defer` 和 `async` 的区别？

这两个属性都是用来控制脚本加载和执行的时机，解决脚本阻塞页面渲染的问题。

**默认情况**下，浏览器遇到 `script` 标签会暂停 HTML 解析，下载并执行完脚本后再继续解析后面的 HTML，这会造成页面渲染的阻塞。

**`async`** 表示异步加载。浏览器会并行下载脚本，下载完成后立即暂停 HTML 解析去执行脚本，执行完再继续解析 HTML。多个 `async` 脚本的执行顺序是不确定的，谁先下载完谁先执行。

**`defer`** 表示延迟执行。浏览器会并行下载脚本，但会等到 HTML 全部解析完成后再按顺序执行脚本。多个 `defer` 脚本会按照它们在文档中出现的顺序执行。

**选择建议**：如果脚本之间没有依赖关系，用 `async`；如果脚本有依赖关系或者需要操作 DOM，用 `defer`。

---

### 1.5 `meta` 标签有哪些常见用途？

`meta` 标签用来描述页面的元信息，不会显示在页面上，但对浏览器、搜索引擎和社交媒体很重要。

**字符编码**：`<meta charset="UTF-8">`，告诉浏览器用 UTF-8 编码解析页面，防止中文乱码。

**视口设置**：`<meta name="viewport" content="width=device-width, initial-scale=1.0">`，这是移动端适配的关键，让页面宽度等于设备宽度，初始缩放比例为 1。

**SEO 相关**：
- `description`：页面描述，会显示在搜索结果摘要中
- `keywords`：关键词（现在搜索引擎基本不看了）
- `robots`：控制搜索引擎是否索引该页面

**HTTP 头部模拟**：
- `http-equiv="refresh"`：定时刷新或跳转
- `http-equiv="X-UA-Compatible"`：指定 IE 的渲染模式

**Open Graph 社交分享**：`og:title`、`og:description`、`og:image` 等，控制分享到微信、微博、Twitter 时的卡片展示。

## 二、DOM 与表单

### 2.1 `DOMContentLoaded` 和 `load` 事件的区别？

这两个事件都是页面加载完成时触发的，但时机不同。

**`DOMContentLoaded`** 在 HTML 文档被完全解析、DOM 树构建完成后触发，不需要等待图片、样式表、iframe 等外部资源加载完成。如果脚本放在 `head` 中且没有 `defer`/`async`，会阻塞这个事件的触发。

**`load`** 在页面所有资源（图片、CSS、JS、字体等）都加载完成后触发，时机更晚。

实际开发中，如果只需要操作 DOM，监听 `DOMContentLoaded` 就够了，响应更快。如果需要获取图片尺寸等资源相关的信息，才需要等 `load` 事件。

---

### 2.2 什么是 DOM 事件流？

DOM 事件流描述的是事件在 DOM 树中传播的过程，分为三个阶段：

**捕获阶段**：事件从 `window` 开始，沿着 DOM 树向下传播，经过各级祖先节点，直到到达目标元素。

**目标阶段**：事件到达实际触发事件的元素。

**冒泡阶段**：事件从目标元素开始，沿着 DOM 树向上冒泡，经过各级祖先节点，直到 `window`。

默认情况下，我们用 `addEventListener` 注册的事件监听器是在冒泡阶段触发的。如果第三个参数传 `true`，就在捕获阶段触发。

实际开发中，大部分场景用冒泡阶段就够了。捕获阶段的一个典型应用是在某些框架中做事件代理。

---

### 2.3 表单验证有哪些方式？

HTML5 提供了**原生表单验证**，不需要写 JS 就能实现基本的验证：

- `required`：字段必填
- `type="email"`：自动验证邮箱格式
- `type="url"`：自动验证 URL 格式
- `minlength` / `maxlength`：限制输入长度
- `min` / `max`：限制数值范围
- `pattern`：用正则表达式自定义验证规则

如果原生验证不够用，可以用 **JavaScript 自定义验证**：

- 监听 `input` 或 `blur` 事件，实时或失焦时验证
- 使用 `setCustomValidity()` 方法设置自定义错误提示
- 结合正则表达式做复杂验证

实际项目中，通常会结合原生验证和 JS 验证，原生验证做第一道防线，JS 验证做更复杂的业务逻辑校验。

---

### 2.4 `localStorage`、`sessionStorage` 和 `Cookie` 的区别？

这三个都是客户端存储方案，但适用场景不同。

| 特性 | `localStorage` | `sessionStorage` | `Cookie` |
|------|---------------|------------------|----------|
| 生命周期 | 永久保存，除非手动清除 | 页面会话结束就清除 | 可设置过期时间 |
| 存储容量 | 约 5~10 MB | 约 5~10 MB | 约 4 KB |
| 作用域 | 同源窗口共享 | 同一窗口/标签页 | 可设置 domain 和 path |
| 服务端读取 | 不能 | 不能 | 每次请求自动携带 |
| 安全性 | 纯文本，可被 XSS 读取 | 纯文本 | 可设置 HttpOnly、Secure |

**使用建议**：
- 需要长期保存的本地数据用 `localStorage`，比如用户偏好设置
- 临时数据用 `sessionStorage`，比如表单草稿
- 需要服务端读取的数据用 `Cookie`，比如登录态 Token（配合 HttpOnly 防 XSS）

## 三、性能与优化

### 3.1 如何优化页面加载速度？

页面加载优化可以从资源、网络、渲染三个层面入手。

**资源层面**：
- 压缩 HTML、CSS、JS，去掉空格和注释
- 图片压缩，使用 WebP 格式，小图标用 SVG 或图标字体
- 代码分割，按需加载，首屏只加载必要资源
- Tree Shaking 去掉未使用的代码

**网络层面**：
- 静态资源上 CDN，利用边缘节点加速
- 开启 Gzip 或 Brotli 压缩
- 使用 HTTP/2 多路复用，减少连接开销
- 合理设置缓存策略，利用浏览器缓存

**渲染层面**：
- CSS 放 `head` 中，避免渲染阻塞
- JS 放 `body` 底部或使用 `defer`/`async`
- 关键 CSS 内联，非关键 CSS 异步加载
- 图片懒加载，首屏外的图片等滚动到附近再加载

---

### 3.2 什么是重排（Reflow）和重绘（Repaint）？如何减少？

**重排**也叫回流，是当元素的几何属性（大小、位置）发生变化时，浏览器需要重新计算布局的过程。重排的成本很高，因为它会影响整个渲染树。

**重绘**是当元素的外观属性（颜色、背景、阴影等）发生变化时，浏览器只需要重新绘制该元素，不需要重新计算布局。重绘的成本比重排低。

**触发重排的常见操作**：修改元素宽高、位置、边距、内容变化、字体大小变化、窗口大小变化等。

**触发重绘的常见操作**：修改颜色、背景、边框样式、可见性等。

**减少重排重绘的方法**：
- 批量修改样式，用 `class` 一次性替换，而不是逐个修改属性
- 对频繁操作的元素使用 `position: absolute` 或 `fixed`，脱离文档流
- 使用 `transform` 和 `opacity` 做动画，这两个属性可以触发 GPU 加速，不会引起重排
- 用 `documentFragment` 或 `display: none` 批量操作 DOM 后再显示
- 用 `will-change` 提前告知浏览器哪些属性会变化

---

### 3.3 什么是懒加载？如何实现？

懒加载就是**延迟加载非首屏资源**，等用户滚动到资源附近时才加载，减少首屏加载时间和带宽消耗。

**图片懒加载**的实现方式：

原生方式是用 `loading="lazy"` 属性，这是 HTML5 原生支持的：

```html
<img src="placeholder.jpg" data-src="real-image.jpg" loading="lazy" alt="描述">
```

兼容旧浏览器可以用 Intersection Observer API：

```javascript
const observer = new IntersectionObserver((entries) => {
  entries.forEach((entry) => {
    if (entry.isIntersecting) {
      const img = entry.target;
      img.src = img.dataset.src; // 把真实地址赋给 src
      observer.unobserve(img);   // 加载后取消观察
    }
  });
});
document.querySelectorAll('img[data-src]').forEach((img) => observer.observe(img));
```

**路由懒加载**在单页应用中也很常用，用 `import()` 动态导入组件，用户访问某个路由时才加载对应的代码块。

## 四、进阶问题

### 4.1 什么是 Shadow DOM？有什么应用场景？

Shadow DOM 是 Web Components 技术的一部分，它允许创建一个**封装的、独立的 DOM 子树**，与页面的主 DOM 隔离。

Shadow DOM 的特点是样式隔离。Shadow DOM 内部的 CSS 不会影响外部，外部的 CSS 也不会影响内部。这解决了组件样式冲突的问题。

**应用场景**：
- 开发可复用的 Web Components，比如自定义视频播放器、日期选择器
- 浏览器内置组件的实现，比如 `video` 标签的控制条就是用 Shadow DOM 实现的
- 第三方组件库（如 Lit、Stencil）底层都用 Shadow DOM 做封装

---

### 4.2 什么是 Web Components？

Web Components 是浏览器原生支持的组件化方案，不需要依赖任何框架，由三个核心技术组成：

**Custom Elements**：允许开发者定义自定义 HTML 标签，比如 `<my-button>`，并封装自己的行为和样式。

**Shadow DOM**：提供 DOM 和样式的封装隔离，防止组件内部和外部互相影响。

**HTML Templates**：用 `<template>` 和 `<slot>` 定义可复用的 HTML 模板结构。

Web Components 的优势是框架无关，任何框架都可以使用。缺点是开发体验不如 React、Vue 等现代框架，生态也不够丰富。目前主要用于开发底层通用组件或跨框架共享的组件。

---

### 4.3 什么是渐进式渲染（Progressive Rendering）？

渐进式渲染是一种**优先渲染关键内容、逐步加载非关键内容**的策略，让用户尽快看到有用的信息。

**具体做法**：
- 服务端先返回 HTML 骨架，浏览器立即渲染出页面框架
- 关键 CSS 内联在 HTML 中，保证首屏样式正确
- 非关键 CSS 异步加载
- 首屏内容优先渲染，首屏外的内容延迟加载
- 图片懒加载，滚动到附近再加载
- 骨架屏（Skeleton Screen）在数据加载前展示占位效果

这种方式比等所有资源加载完再一次性渲染用户体验更好，用户不会面对白屏等待。

---

### 4.4 什么是 Service Worker？能做什么？

Service Worker 是运行在浏览器后台的独立线程，可以拦截和处理网络请求，实现离线缓存、推送通知、后台同步等功能。

**核心能力**：
- **离线缓存**：把静态资源缓存到本地，没网时也能访问
- **网络请求拦截**：可以自定义缓存策略，比如优先读缓存、优先发请求、或者两者结合
- **推送通知**：即使用户关闭了页面，也能收到推送消息
- **后台同步**：网络恢复后自动同步之前失败的操作

Service Worker 的生命周期包括安装（install）、激活（activate）和运行（fetch/message）三个阶段。它不能直接操作 DOM，需要通过 `postMessage` 和页面通信。

PWA（渐进式 Web 应用）的核心技术之一就是 Service Worker，它让 Web 应用可以像原生 App 一样离线使用。

---

### 4.5 前端安全方面需要注意哪些问题？

前端安全主要关注 **XSS** 和 **CSRF** 两类攻击。

**XSS（跨站脚本攻击）** 是攻击者往页面注入恶意脚本，窃取用户信息或执行恶意操作。防御手段包括：对用户输入做 HTML 转义、设置 CSP 安全策略、Cookie 设置 HttpOnly。

**CSRF（跨站请求伪造）** 是攻击者诱导用户在已登录的网站上执行非预期操作。防御手段包括：使用 CSRF Token、设置 Cookie 的 SameSite 属性、验证请求来源。

**其他安全措施**：
- HTTPS 防止中间人攻击
- 敏感操作做二次确认
- 前端代码混淆压缩，减少被分析的风险
- 定期进行安全扫描和依赖漏洞检测
