# 从下载注册开始使用 Codex

## 适合谁看

这篇文档面向第一次接触 Codex 的开发者。目标是从账号准备、安装客户端、登录、打开项目，到第一次让 Codex 修改代码，走完一条完整路径。

如果你已经能正常使用 Codex，可以直接跳到“第一次使用建议”和“常见问题”。

## 先理解 Codex 的几种入口

Codex 不是只有一个入口。常见使用方式有：

- Codex App：图形界面，适合在本地项目中协作开发。
- Codex CLI：命令行工具，适合终端用户。
- Codex IDE Extension：编辑器扩展，适合在 IDE 内直接使用。
- Codex Web / Cloud：云端任务入口，适合让 Codex 在远程环境里处理代码任务。

日常开发里，推荐优先从 Codex App 或 Codex CLI 开始。它们最贴近本地项目开发流程。

## 第一步：注册或准备 ChatGPT 账号

### 1. 打开 ChatGPT

访问：

```txt
https://chatgpt.com
```

如果你还没有账号，先注册。可以使用邮箱、Google、Microsoft、Apple 等方式，具体以页面显示为准。

### 2. 完成登录

登录后确认账号可以正常进入 ChatGPT。Codex 可以通过 ChatGPT 账号登录，具体可用额度和功能入口会根据你的账号计划而不同。

### 3. 确认计划和权限

Codex 支持多个 ChatGPT 计划，但不同计划的额度、功能入口、组织策略可能不同。个人账号通常直接登录即可；团队、企业、教育账号可能会受到管理员策略影响。

建议确认：

- 当前账号能否访问 Codex。
- 是否允许使用本地 Codex。
- 是否允许连接 GitHub。
- 是否允许使用云端任务。

## 第二步：选择安装方式

## 方式一：使用 Codex App

Codex App 更适合图形化使用。你可以直接打开项目、查看文件、运行命令、看页面效果，并和 Codex 对话。

推荐流程：

1. 下载并安装 Codex App。
2. 打开应用。
3. 使用 ChatGPT 账号登录。
4. 选择本地项目目录。
5. 让 Codex 阅读项目并执行任务。

第一次打开项目后，可以先问：

```txt
分析这个项目的技术栈和目录结构，不要修改代码。
```

确认 Codex 能正确读取项目后，再开始让它修改代码。

## 方式二：使用 Codex CLI

Codex CLI 适合习惯命令行的开发者。官方帮助文档中的安装方式是：

```bash
npm install -g @openai/codex
```

安装完成后，在终端执行：

```bash
codex
```

然后根据终端提示登录 ChatGPT 账号。

如果之前通过 API Key 使用过 Codex CLI，建议先升级 CLI，再重新登录：

```bash
codex --upgrade
codex logout
codex
```

## 第三步：准备本地开发环境

Codex 能帮你修改代码，但项目本身仍然需要可运行的本地环境。

常见准备项：

- Node.js
- npm / pnpm / yarn
- Git
- 项目依赖
- 本地端口可用
- 浏览器

以前端项目为例，通常先执行：

```bash
npm install
npm start
```

如果项目已经有 `node_modules`，可以先直接启动：

```bash
npm start
```

## 第四步：打开项目

进入项目目录后再启动 Codex。

示例：

```bash
cd D:\work\resourse\myself
codex
```

如果使用 Codex App，则直接选择项目目录。

打开项目后，第一句话建议这样问：

```txt
先分析这个项目结构，说明主要页面、路由和启动方式，不要修改代码。
```

这样可以先确认 Codex 对项目的理解是否正确。

## 第五步：第一次让 Codex 做任务

第一次不要给太大的任务。推荐从小问题开始。

示例 1：启动项目

```txt
启动这个前端项目，并告诉我访问地址。
```

示例 2：分析页面

```txt
分析首页当前结构，给出后续优化建议，不要修改代码。
```

示例 3：修一个明确 bug

```txt
模型预览页第一次进入没有默认模型，请修复，并验证页面能正常显示。
```

示例 4：优化一个模块

```txt
优化 Docs 模块，添加搜索、分类、目录和代码复制按钮。
```

## 第六步：给 Codex 明确约束

Codex 可以做很多事，所以更需要边界。

常用约束：

```txt
只改 src/pages/docs 相关文件。
```

```txt
不要升级依赖。
```

```txt
不要删除用户已有改动。
```

```txt
改完后只跑 npm run build。
```

```txt
如果需要大范围重构，先说明方案再动手。
```

## 第七步：验收 Codex 的结果

每次改完至少检查三件事：

1. 代码是否符合需求。
2. 构建或测试是否通过。
3. 页面实际效果是否正常。

前端项目建议加上：

```txt
打开浏览器检查控制台是否有 error。
```

如果是 UI 改动，建议同时检查：

- 桌面端
- 移动端
- 首屏遮挡
- 文字溢出
- 按钮点击

## 常见问题

## npm install -g 失败

可能原因：

- Node.js 没安装
- npm 版本太旧
- 网络访问失败
- 全局安装目录没有权限

可以先检查：

```bash
node -v
npm -v
```

如果是权限问题，可以换 Node 版本管理工具，或者调整 npm 全局目录。

## codex 命令找不到

可能是全局 npm bin 目录没有加入环境变量。

检查：

```bash
npm bin -g
```

然后确认这个目录是否在 `PATH` 中。

## 登录失败

可能原因：

- 浏览器没有完成登录
- 当前账号不支持对应入口
- 组织策略禁用了 Codex
- 网络代理或防火墙拦截

处理方式：

1. 重新执行 `codex logout`。
2. 再执行 `codex`。
3. 使用浏览器完成 ChatGPT 登录。
4. 如果是团队账号，确认管理员是否开放 Codex。

## Codex 不敢执行命令

这是正常的权限保护。Codex 通常会在高风险操作前请求确认。

常见高风险操作：

- 删除文件
- 重置 Git
- 安装依赖
- 访问网络
- 修改工作区外文件

如果你确定可以执行，可以明确授权。

## Codex 改动太大

下次可以这样约束：

```txt
只做最小改动，不要重构无关代码。
```

或者：

```txt
先列方案，不要直接修改。
```

## 推荐的第一次练习

建议按这个顺序练习：

1. 让 Codex 分析项目。
2. 让 Codex 启动项目。
3. 让 Codex 修一个明确 bug。
4. 让 Codex 优化一个小组件。
5. 让 Codex 写一篇项目文档。
6. 让 Codex 做一次完整模块优化。

## 安全建议

使用 Codex 时要注意：

- 不要把真实密钥写进对话。
- 不要让 Codex 随意提交生产代码。
- 删除、重置、覆盖文件前先确认。
- 涉及账号、支付、生产数据时要人工复核。
- 对不确定的工具命令先问清楚作用。
